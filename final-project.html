TFT-gogo 전적 검색 서비스

전략적 팀 전투(TFT) 유저가 소환사 전적을 검색하고 최근 매치, 증강, 덱 구성, 메타덱과 게임 가이드를 확인할 수 있는 서비스입니다. Spring Boot 백엔드, React 프론트엔드, FastAPI AI 서버를 모노레포로 구성하고 Riot API와 OpenAI API 연동 흐름을 함께 다루고 있습니다.

Design Assets

설계 산출물

Features

구현 기능

소환사 전적 검색과 매치 분석

Riot API에서 소환사와 매치 데이터를 조회해 최근 경기, 순위, 증강, 챔피언·아이템 구성 정보를 화면에 제공합니다. 외부 API 응답 실패와 데이터 없음 상태를 구분해 사용자 흐름이 끊기지 않도록 설계했습니다.

메타덱 목록

최근 메타 데이터를 기준으로 덱 구성, 승률, 평균 등수, 픽률을 한 화면에서 비교할 수 있도록 구성했습니다. 반복 조회되는 메타 정보는 DB와 캐시 전략을 함께 고려해 불필요한 외부 호출을 줄이는 방향으로 설계했습니다.

게임 가이드

시너지, 아이템, 증강체, 챔피언 정보를 한 화면에서 빠르게 비교할 수 있도록 구성했습니다. Community Dragon 자산을 활용해 게임 데이터와 이미지 리소스를 분리하고, 기준 패치에 맞는 가이드 정보를 조회할 수 있게 했습니다.

AI 맞춤 덱 추천

최근 전적을 기반으로 자주 플레이한 시너지와 성적을 분석하고, 현재 메타 데이터와 함께 비교해 추천 덱을 제안하는 흐름을 구성했습니다. Spring Boot API와 FastAPI AI 서버의 책임을 나눠 추천 요청과 응답 경계를 분리했습니다.

Architecture Notes

구현하면서 고려한 기술 포인트

TFT-gogo는 Riot API, Community Dragon, AI 서버, DB 캐시처럼 여러 외부·내부 의존성이 맞물립니다. 그래서 요청이 어디서 시작되고 어느 저장소와 외부 API에 도달하는지 추적 가능한 구조를 우선했습니다.

Backend API

프론트엔드와 맞출 API 계약을 먼저 정리했습니다

소환사 검색, 매치 목록, 메타덱, 게임 가이드, AI 추천 기능은 요청·응답 필드와 실패 응답 형식을 먼저 맞췄습니다. Spring Boot에서는 Controller, Service, Repository 책임을 나누고 BusinessException 기반 공통 예외로 프론트엔드가 같은 방식으로 처리할 수 있게 했습니다.

Java Spring Boot JPA BusinessException REST API

Frontend State

검색·상세·목록 화면의 상태가 어긋나지 않게 나눴습니다

React 화면에서는 검색 입력, 최근 검색, 로딩, 데이터 없음, 실패 상태를 구분했습니다. Riot API 결과가 늦거나 실패해도 이전 화면 상태와 섞이지 않도록 API 응답 기준으로 렌더링 조건을 나누고, 사용자에게 보여줄 메시지를 화면별로 정리했습니다.

React TypeScript Axios Loading State Error State

AI & Data

AI 추천 서버와 데이터 저장소의 책임을 분리했습니다

AI 추천은 Spring Boot API가 요청을 받고 FastAPI 서버가 추천 로직을 처리하는 구조로 나눴습니다. 메타덱·게임 가이드처럼 반복 조회되는 데이터는 MySQL과 Redis 캐시를 고려했고, 추천 데이터는 PostgreSQL/pgvector와 OpenAI API를 함께 활용하는 흐름을 학습하며 구성했습니다.

FastAPI OpenAI API MySQL Redis PostgreSQL/pgvector

Infra & Integration

모노레포 서비스를 한 번에 실행하고 연결 상태를 확인했습니다

Docker Compose로 React, Spring Boot, FastAPI, MySQL, Redis, PostgreSQL/pgvector를 함께 실행할 수 있도록 구성했습니다. 서비스 간 연결 실패가 발생했을 때 API 주소, 내부 시크릿, DB URL, 컨테이너 상태 중 어디를 확인해야 하는지 기준을 나눴습니다.

Docker Docker Compose AWS ECS ECR ALB
API 계약

프론트엔드와 맞춰야 하는 소환사 검색, 매치 목록, 메타덱, 게임 가이드, AI 추천 API의 요청·응답 형식을 먼저 정리했습니다. 실패 응답도 같은 공통 포맷으로 다룰 수 있도록 필드를 명확히 나눴습니다.

외부 API 경계

Riot API 키, 호출 제한, 응답 지연, 데이터 없음 상태를 고려해 백엔드에서 실패 조건을 구분했습니다. 외부 API 결과를 그대로 화면에 넘기지 않고 서비스 응답으로 정리해 프론트엔드가 안정적으로 처리하도록 했습니다.

집계와 캐시

메타덱 집계와 게임 가이드 자산처럼 반복 조회되는 데이터는 DB와 Redis, 캐시 전략을 함께 고려했습니다. 최신 데이터 반영과 불필요한 외부 호출 감소 사이의 균형을 잡는 데 집중했습니다.

로컬 통합 실행

Docker Compose로 MySQL, Redis, PostgreSQL/pgvector, Spring Boot, FastAPI, React를 함께 띄울 수 있도록 구성했습니다. 서비스 간 연결 실패가 발생했을 때 어느 컨테이너와 환경 변수를 확인해야 하는지 정리했습니다.

Release Management

팀 협업과 릴리즈 관리

TFT-gogo는 여러 사람이 동시에 작업하는 팀 프로젝트로 진행했습니다. 요구사항 문서, API 계약, 이슈와 PR, 배포 체크리스트, 릴리즈 이후 확인 기준을 연결해 기능 구현이 실제 실행 가능한 상태까지 이어지도록 관리했습니다.

01 / Requirement

기능 범위와 핵심 목표를 먼저 정리했습니다

프로젝트 초기에 기능 영역별 목표와 구현 범위를 문서로 정리했습니다. 전적 검색, 메타 분석, AI 추천, 게임 가이드처럼 의존성이 다른 기능을 나누고 각 기능이 어떤 사용자 흐름과 데이터 기준을 가져야 하는지 먼저 맞췄습니다.

  • 관리 항목 기능 영역, 구현 기능, 핵심 목표, 우선순위
  • 확인 방식 기능 단위로 범위를 나누고 이슈와 PR에서 구현 기준을 확인

02 / Issue & PR

이슈와 PR을 라벨로 분류해 변경 이력을 남겼습니다

작업은 GitHub Issue로 분리하고 backend, frontend, infra, bug, perf 같은 라벨로 성격을 구분했습니다. PR에서는 관련 이슈를 연결해 머지와 동시에 이슈가 닫히도록 관리했고, 구현 의도와 확인 방법을 남겨 변경 이력을 추적할 수 있게 했습니다.

  • 관리 항목 Issue, PR, Label, Milestone, 리뷰 코멘트
  • 확인 방식 PR 설명에 변경 요약과 테스트 계획을 남기고 관련 이슈를 연결

03 / API & Integration

API와 외부 연동 범위를 기능별로 나눴습니다

Riot API, Community Dragon, FastAPI 추천 서버, 인증/회원 기능처럼 필요한 데이터와 외부 연동이 다른 영역을 기능별로 분리했습니다. 구현 전에 어떤 API와 데이터가 필요한지 정리해 프론트엔드와 백엔드 작업 기준이 어긋나지 않도록 했습니다.

  • 관리 항목 Riot API, Community Dragon, FastAPI, 인증 API, AI 추천 데이터
  • 확인 방식 기능별 필요 API와 사용 목적을 문서화한 뒤 구현 범위 확인

04 / Release

릴리즈 단위로 변경 사항과 후속 작업을 정리했습니다

기능 추가, 버그 수정, 성능 개선, 리팩토링, 문서 작업을 릴리즈 노트에 묶어 기록했습니다. 각 릴리즈마다 포함된 PR과 변경 내용을 남기고, 완료되지 않은 작업은 다음 마일스톤으로 넘겨 이후 개발 범위가 흐려지지 않도록 관리했습니다.

  • 관리 항목 Release Note, PR 목록, 버그 수정, 성능 개선, 다음 버전 전환 항목
  • 확인 방식 릴리즈별 변경 사항과 후속 과제를 분리해 기록

05 / AI Skills

AI를 개발 흐름을 보조하는 도구로 활용했습니다

요구사항을 먼저 정리하고, AI에게 역할과 제약 조건, 검증 기준을 함께 전달해 코드 초안을 만들었습니다. 생성된 코드는 바로 붙여넣지 않고 프로젝트 구조, 의존성, 예외 처리 방식에 맞는지 읽고 수정한 뒤 적용했습니다.

  • 관리 항목 스펙 정리, 프롬프트 컨텍스트, 코드 리뷰, 검증 체크
  • 확인 방식 AI 제안 코드를 분석하고 팀 컨벤션과 검증 기준에 맞게 조정
