생성형 AI 기반 SaaS 업무 지원 플랫폼

조직 단위 사용자를 대상으로 문서 업로드, AI 요약, 질의응답, 권한 관리, 사용 이력 조회를 제공하는 SaaS 플랫폼입니다. 팀 프로젝트로 진행하며 요구사항 정의, API 설계, 기능 구현, 배포 확인, 협업 문서화를 함께 경험했습니다.

Design Assets

Features

구현 기능

조직과 사용자 권한 관리

조직 단위로 사용자를 초대하고 역할을 부여할 수 있도록 설계했습니다. 일반 사용자와 관리자 권한을 구분해 접근 가능한 API와 화면을 나눴습니다.

문서 업로드와 AI 요약

사용자가 업로드한 문서를 저장하고 AI 요약 요청을 생성합니다. 요청 상태를 저장해 처리 중, 완료, 실패 상태를 화면에서 확인할 수 있도록 구성했습니다.

문서 기반 질의응답

문서와 질문을 연결해 답변을 생성하는 흐름을 구현했습니다. 응답 이력을 남겨 사용자가 이전 질문과 답변을 다시 확인할 수 있도록 했습니다.

사용 이력과 관리자 대시보드

조직별 요청 수, 사용자별 사용 이력, 최근 오류를 확인할 수 있도록 대시보드를 구성했습니다. 운영 관점에서 사용량과 실패 흐름을 추적할 수 있게 했습니다.

Architecture Notes

구현하면서 고려한 기술 포인트

기능이 늘어날수록 인증, 권한, 외부 API 호출, 배포 설정이 서로 영향을 줍니다. 그래서 책임을 나누고 실패 지점을 확인할 수 있는 구조를 우선했습니다.

API 계약

프론트엔드와 맞춰야 하는 요청/응답, 에러 포맷을 먼저 정리했습니다. 문서 업로드, AI 요약 요청, 질의응답, 관리자 조회처럼 화면과 맞닿는 기능은 응답 구조가 흔들리지 않도록 필드를 명확히 나눴습니다.

권한 경계

조직 관리자와 일반 사용자의 접근 가능 범위를 API 계층에서 확인했습니다. 같은 문서라도 소속 조직과 역할에 따라 접근이 달라질 수 있어 조회와 수정 요청마다 권한 검증 흐름을 분리했습니다.

비동기 처리

AI 응답 지연을 고려해 요청 상태를 저장하고 사용자에게 처리 상태를 보여줬습니다. 요청 생성, 처리 중, 완료, 실패 상태를 구분해 긴 작업이 화면 흐름을 막지 않도록 나눴습니다.

운영 확인

배포 후 확인해야 할 로그, 환경 변수, 외부 API 연결 실패 상황을 체크했습니다. 오류가 발생했을 때 어느 구간에서 실패했는지 추적할 수 있도록 요청 흐름과 설정 값을 문서로 남겼습니다.

Release Management

팀 협업과 릴리즈 관리

파이널프로젝트는 여러 사람이 동시에 작업하는 팀 프로젝트로 진행했습니다. 요구사항 문서, API 계약, 이슈와 PR, 배포 체크리스트, 릴리즈 이후 확인 기준을 연결해 기능 구현이 실제 실행 가능한 상태까지 이어지도록 관리했습니다.

01 / Requirement

Notion에서 요구사항과 역할을 먼저 맞췄습니다

사용자 역할, 핵심 화면, API 필요 항목, 담당 범위를 문서로 정리했습니다. 변경된 요구사항은 구현 전에 문서에 먼저 반영해 팀원이 같은 기준을 보고 작업하도록 맞췄습니다.

  • 관리 항목 회의록, 화면 정의, 역할 분담, 완료 기준
  • 확인 방식 작업 전 문서 기준을 맞추고 이슈로 구현 범위를 분리

02 / Pull Request

Issue, Branch, PR을 연결해 변경 이력을 남겼습니다

GitHub Issue를 기준으로 브랜치를 만들고 PR에서 구현 의도, 변경 범위, 확인 방법을 공유했습니다. 리뷰 코멘트가 생기면 수정 커밋으로 반영해 논의가 코드에 어떻게 반영됐는지 추적할 수 있게 했습니다.

  • 관리 항목 브랜치 전략, 커밋 컨벤션, PR 설명, 코드리뷰 코멘트
  • 확인 방식 병합 전 변경 파일과 테스트 결과를 PR에 함께 기록

03 / Release

API 계약과 배포 준비로 릴리즈 위험을 줄였습니다

요청/응답 필드, 에러 포맷, 인증이 필요한 API를 프론트엔드와 먼저 맞췄습니다. 배포 전에는 환경 변수, DB 연결, 외부 AI API 키, 빌드 결과를 확인해 릴리즈 직전의 실패 가능성을 줄였습니다.

  • 관리 항목 API 계약, 환경 변수, DB 연결, 외부 API 키, 빌드 결과
  • 확인 방식 배포 전 체크리스트로 누락 항목을 먼저 제거

04 / Release QA

릴리즈 이후 확인 기준과 대응 순서를 정리했습니다

배포가 끝난 뒤에는 주요 화면 접근, 로그인, 문서 업로드, AI 요청 생성, 관리자 조회처럼 사용자가 실제로 거치는 흐름을 기준으로 다시 확인했습니다. 문제가 발생했을 때는 로그, 환경 변수, DB 연결, 외부 API 응답을 순서대로 확인하는 기준을 팀 문서에 남겼습니다.

  • 관리 항목 릴리즈 노트, 배포 후 확인 목록, 장애 대응 순서
  • 확인 방식 핵심 사용자 흐름과 실패 로그를 기준으로 릴리즈 완료 판단

05 / AI Skills

AI를 개발 흐름을 보조하는 도구로 활용했습니다

요구사항을 먼저 정리하고, AI에게 역할과 제약 조건, 검증 기준을 함께 전달해 코드 초안을 만들었습니다. 생성된 코드는 바로 붙여넣지 않고 프로젝트 구조, 의존성, 예외 처리 방식에 맞는지 읽고 수정한 뒤 적용했습니다.

  • 관리 항목 스펙 정리, 프롬프트 컨텍스트, 코드 리뷰, 검증 체크
  • 확인 방식 AI 제안 코드를 분석하고 팀 컨벤션과 검증 기준에 맞게 조정
