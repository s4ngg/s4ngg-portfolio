TFT-gogo 전적 검색 서비스

전략적 팀 전투(TFT) 유저가 소환사 전적을 검색하고 최근 매치, 증강, 덱 구성, 메타덱, 패치노트를 확인할 수 있는 서비스입니다. Spring Boot 백엔드, React 프론트엔드, FastAPI AI 서버를 모노레포로 구성하고 Riot API와 OpenAI API 연동 흐름을 함께 다루고 있습니다.

Features

구현 기능

소환사 전적 검색과 매치 분석

Riot API에서 소환사와 매치 데이터를 조회해 최근 경기, 순위, 증강, 챔피언·아이템 구성 정보를 화면에 제공합니다. 외부 API 응답 실패와 데이터 없음 상태를 구분해 사용자 흐름이 끊기지 않도록 설계했습니다.

메타덱과 AI 추천

메타덱 데이터를 수집·집계하고 FastAPI AI 서버와 연동해 현재 상황에 맞는 덱 추천 흐름을 구성했습니다. 추천 요청은 백엔드 API를 거쳐 AI 서버로 전달되도록 경계를 나눴습니다.

게임 가이드와 패치노트

챔피언, 시너지, 증강 정보와 패치노트를 정리해 사용자가 메타 변화를 빠르게 확인할 수 있도록 구성했습니다. Community Dragon 자산과 패치노트 데이터를 API로 분리했습니다.

커뮤니티와 파티 모집

유저가 공략을 공유하고 파티를 모집할 수 있는 커뮤니티 기능을 제공합니다. 채팅과 파티 상태는 화면 상태와 서버 상태가 어긋나지 않도록 API 응답 기준을 맞췄습니다.

Architecture Notes

구현하면서 고려한 기술 포인트

TFT-gogo는 Riot API, Community Dragon, AI 서버, DB 캐시처럼 여러 외부·내부 의존성이 맞물립니다. 그래서 요청이 어디서 시작되고 어느 저장소와 외부 API에 도달하는지 추적 가능한 구조를 우선했습니다.

API 계약

프론트엔드와 맞춰야 하는 소환사 검색, 매치 목록, 메타덱, 패치노트, 파티 모집 API의 요청·응답 형식을 먼저 정리했습니다. 실패 응답도 같은 공통 포맷으로 다룰 수 있도록 필드를 명확히 나눴습니다.

외부 API 경계

Riot API 키, 호출 제한, 응답 지연, 데이터 없음 상태를 고려해 백엔드에서 실패 조건을 구분했습니다. 외부 API 결과를 그대로 화면에 넘기지 않고 서비스 응답으로 정리해 프론트엔드가 안정적으로 처리하도록 했습니다.

집계와 캐시

메타덱 집계, 가이드 자산, 패치노트처럼 반복 조회되는 데이터는 DB와 Redis, 캐시 전략을 함께 고려했습니다. 최신 데이터 반영과 불필요한 외부 호출 감소 사이의 균형을 잡는 데 집중했습니다.

로컬 통합 실행

Docker Compose로 MySQL, Redis, PostgreSQL/pgvector, Spring Boot, FastAPI, React를 함께 띄울 수 있도록 구성했습니다. 서비스 간 연결 실패가 발생했을 때 어느 컨테이너와 환경 변수를 확인해야 하는지 정리했습니다.

Release Management

팀 협업과 릴리즈 관리

파이널프로젝트는 여러 사람이 동시에 작업하는 팀 프로젝트로 진행했습니다. 요구사항 문서, API 계약, 이슈와 PR, 배포 체크리스트, 릴리즈 이후 확인 기준을 연결해 기능 구현이 실제 실행 가능한 상태까지 이어지도록 관리했습니다.

01 / Requirement

Notion에서 요구사항과 역할을 먼저 맞췄습니다

사용자 역할, 핵심 화면, API 필요 항목, 담당 범위를 문서로 정리했습니다. 변경된 요구사항은 구현 전에 문서에 먼저 반영해 팀원이 같은 기준을 보고 작업하도록 맞췄습니다.

  • 관리 항목 회의록, 화면 정의, 역할 분담, 완료 기준
  • 확인 방식 작업 전 문서 기준을 맞추고 이슈로 구현 범위를 분리

02 / Pull Request

Issue, Branch, PR을 연결해 변경 이력을 남겼습니다

GitHub Issue를 기준으로 브랜치를 만들고 PR에서 구현 의도, 변경 범위, 확인 방법을 공유했습니다. 리뷰 코멘트가 생기면 수정 커밋으로 반영해 논의가 코드에 어떻게 반영됐는지 추적할 수 있게 했습니다.

  • 관리 항목 브랜치 전략, 커밋 컨벤션, PR 설명, 코드리뷰 코멘트
  • 확인 방식 병합 전 변경 파일과 테스트 결과를 PR에 함께 기록

03 / Release

API 계약과 배포 준비로 릴리즈 위험을 줄였습니다

요청/응답 필드, 에러 포맷, 인증이 필요한 API를 프론트엔드와 먼저 맞췄습니다. 배포 전에는 환경 변수, DB 연결, 외부 AI API 키, 빌드 결과를 확인해 릴리즈 직전의 실패 가능성을 줄였습니다.

  • 관리 항목 API 계약, 환경 변수, DB 연결, 외부 API 키, 빌드 결과
  • 확인 방식 배포 전 체크리스트로 누락 항목을 먼저 제거

04 / Release QA

릴리즈 이후 확인 기준과 대응 순서를 정리했습니다

배포가 끝난 뒤에는 주요 화면 접근, 소환사 검색, 매치 조회, 메타덱 조회, AI 추천 요청, 커뮤니티 기능처럼 사용자가 실제로 거치는 흐름을 기준으로 다시 확인했습니다. 문제가 발생했을 때는 로그, 환경 변수, DB 연결, Riot API·AI 서버 응답을 순서대로 확인하는 기준을 팀 문서에 남겼습니다.

  • 관리 항목 릴리즈 노트, 배포 후 확인 목록, 장애 대응 순서
  • 확인 방식 핵심 사용자 흐름과 실패 로그를 기준으로 릴리즈 완료 판단

05 / AI Skills

AI를 개발 흐름을 보조하는 도구로 활용했습니다

요구사항을 먼저 정리하고, AI에게 역할과 제약 조건, 검증 기준을 함께 전달해 코드 초안을 만들었습니다. 생성된 코드는 바로 붙여넣지 않고 프로젝트 구조, 의존성, 예외 처리 방식에 맞는지 읽고 수정한 뒤 적용했습니다.

  • 관리 항목 스펙 정리, 프롬프트 컨텍스트, 코드 리뷰, 검증 체크
  • 확인 방식 AI 제안 코드를 분석하고 팀 컨벤션과 검증 기준에 맞게 조정
