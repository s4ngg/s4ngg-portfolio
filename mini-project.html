Features

구현 기능

회원·판매자 인증과 역할 분리

일반 회원과 판매자, 관리자 계정을 분리하고 JWT 기반 인증으로 접근 가능한 API를 구분했습니다. 판매자 신청과 승인 상태에 따라 상품 등록 가능 여부가 달라지도록 흐름을 나눴습니다.

상품·카테고리·승인 관리

판매자가 상품을 등록하면 관리자가 승인 또는 반려할 수 있도록 상품 상태를 관리했습니다. 부모·자식 카테고리 구조를 기준으로 상품 목록과 상세 조회가 이어지도록 구성했습니다.

장바구니·주문·결제 흐름

장바구니 담기부터 주문 생성, 배송지 관리, 결제 정보 조회까지 구매 흐름을 연결했습니다. React 화면에서는 API 응답과 사용자 상태를 기준으로 주문 단계를 분리했습니다.

AI 상품 추천과 CS 기능

FastAPI 서버에서 관심 카테고리와 위시리스트를 받아 OpenAI API 기반 추천 결과를 반환하도록 구성했습니다. 문의, 클레임, FAQ, 공지사항은 관리자와 사용자 흐름을 나눠 처리했습니다.

Architecture Notes

구현하면서 고려한 기술 포인트

쇼핑몰은 화면 단위보다 주문, 승인, 결제, CS처럼 상태가 바뀌는 흐름이 중요했습니다. 그래서 역할별 권한과 도메인 책임을 나누고, API 응답과 화면 상태가 어긋나지 않도록 확인했습니다.

도메인 분리

회원, 상품, 주문, 쿠폰, 리뷰, CS, 관리자 기능을 도메인별로 나누고 Controller, Service, Repository 책임을 분리했습니다. 기능을 수정할 때 영향 범위를 좁히는 것을 기준으로 삼았습니다.

예외 처리

BusinessException과 공통 응답 형식을 기준으로 입력 오류, 인증 실패, 권한 실패, 존재하지 않는 리소스를 구분했습니다. 프론트엔드에서 같은 방식으로 실패 메시지를 처리할 수 있도록 응답 계약을 맞췄습니다.

DB 설계

상품, 카테고리, 주문, 쿠폰, 리뷰, 문의·클레임의 관계와 상태 값을 기준으로 테이블 구조를 잡았습니다. 승인·반려·취소처럼 상태가 바뀌는 기능은 후속 조회 결과까지 함께 확인했습니다.

배포 확인

프론트엔드는 S3와 CloudFront, 백엔드는 EC2 배포 흐름을 기준으로 환경 변수와 로그를 점검했습니다. API 주소, Toss Payments, EmailJS, S3, Redis처럼 외부 설정이 필요한 항목을 분리해 정리했습니다.

Verification

기능 구현과 검증 기록

미니 프로젝트에서는 쇼핑몰 기능을 화면별로만 보지 않고 회원, 판매자, 관리자 흐름이 API와 DB 상태로 어떻게 이어지는지 확인했습니다. 인증, 주문, 승인, 추천, CS 기능을 구현 단위로 나누고 실패 조건과 응답 흐름을 함께 기록했습니다.

01 / Authentication

회원·판매자·관리자 인증 기준을 분리했습니다

일반 회원 로그인, 판매자 로그인, 관리자 로그인 흐름을 나누고 요청 경로에 따라 필요한 토큰과 권한이 다르게 적용되도록 확인했습니다.

  • 확인 조건 로그인 실패, 토큰 누락, 역할이 맞지 않는 관리자·판매자 요청
  • 검증 결과 역할별 접근 가능 API와 화면 노출 기준을 분리

02 / Permission

승인과 주문 상태가 뒤틀리지 않도록 확인했습니다

판매자 승인, 상품 승인, 주문 생성, 클레임 처리처럼 상태가 바뀌는 기능은 요청 성공 이후 실제 DB 상태와 후속 조회 결과가 일치하는지 확인했습니다.

  • 확인 조건 승인 전 상품 등록, 존재하지 않는 주문, 이미 처리된 클레임
  • 검증 결과 실패 조건별 메시지와 상태 변경 여부를 문서에 기록

03 / AI & Integration

외부 연동 기능은 입력과 실패 응답을 함께 확인했습니다

Toss Payments, EmailJS, 사업자등록번호 검증, FastAPI 추천 서버처럼 외부 설정이 필요한 기능은 요청 값, 환경 변수, 실패 응답을 분리해 확인했습니다.

  • 확인 조건 추천 입력 누락, 외부 API 키 누락, 결제·메일 연동 실패
  • 검증 결과 기능별 재현 방법과 수정 결과를 이슈 단위로 정리
