회원 기반 게시판 서비스

Design Assets

설계 산출물

Features

구현 기능

회원가입과 로그인

아이디 중복 검증, 비밀번호 암호화, 세션 기반 로그인 상태 유지, 인증 실패 메시지를 구현했습니다. 로그인 여부에 따라 작성, 수정, 삭제 접근 기준도 분리했습니다.

게시글 CRUD와 권한 검증

게시글 작성, 상세 조회, 수정, 삭제를 구현하고 작성자와 관리자만 수정/삭제할 수 있도록 서비스 계층에서 권한을 검증했습니다.

검색과 페이징

제목, 작성자, 내용 기준 검색과 페이지 이동을 구현했습니다. 조회 조건이 늘어나도 흐름을 추적하기 쉽도록 Repository 책임을 분리했습니다.

댓글과 입력 검증

댓글 작성과 삭제, 빈 값 검증, 길이 제한, 예외 응답을 처리했습니다. 잘못된 입력이 들어왔을 때 사용자가 다시 시도할 수 있도록 메시지를 분리했습니다.

Architecture Notes

구현하면서 고려한 기술 포인트

게시판은 단순 CRUD처럼 보이지만 로그인 상태, 작성자 권한, 검색 조건, 댓글 관계가 함께 얽힙니다. 그래서 요청 흐름과 책임을 나누고, 실패 조건을 명확히 처리하는 데 집중했습니다.

도메인 분리

Controller는 요청과 응답에 집중하고, 핵심 규칙은 Service에서 처리하도록 나눴습니다. Repository는 데이터 접근 책임을 맡겨 기능을 수정할 때 영향 범위를 좁히는 것을 기준으로 삼았습니다.

예외 처리

입력 오류, 인증 실패, 권한 실패를 구분해 원인을 파악할 수 있는 응답을 반환했습니다. 사용자가 다시 시도할 수 있는 오류와 개발자가 확인해야 하는 오류를 나눠 기록했습니다.

DB 설계

회원, 게시글, 댓글 간 관계와 삭제 정책을 기준으로 테이블 구조를 잡았습니다. 외래 키 관계와 조회 조건을 고려해 검색과 페이징 기능이 자연스럽게 이어지도록 설계했습니다.

배포 확인

로컬 실행과 AWS 실행 환경의 차이를 확인하며 환경 변수와 로그를 점검했습니다. 실행 방법, 필요한 설정 값, 오류가 발생했을 때 확인할 위치를 README에 정리하는 방식으로 마무리했습니다.

Verification

기능 구현과 검증 기록

개인 프로젝트에서는 기능을 많이 넣는 것보다 각 기능의 책임을 나누고, 정상 동작과 실패 조건을 함께 확인하는 데 집중했습니다. 회원, 게시글, 댓글, 검색 기능을 구현 단위로 분리하고 예외, 권한, 입력 검증 결과를 남겼습니다.

01 / Authentication

회원 인증을 먼저 고정해 접근 기준을 만들었습니다

가입, 로그인, 로그아웃, 인증 실패 흐름을 먼저 구현하고 이후 게시글 작성, 댓글 작성, 수정/삭제 기능의 접근 기준으로 사용했습니다.

  • 확인 조건 아이디 중복, 비밀번호 오류, 비로그인 요청
  • 검증 결과 로그인 상태에 따라 버튼 노출과 접근 가능 경로를 분리

02 / Permission

권한과 예외 조건을 화면 버튼 밖에서도 확인했습니다

작성자와 관리자만 게시글과 댓글을 수정/삭제할 수 있도록 서비스 계층에서 검증했습니다. 화면 버튼 노출뿐 아니라 직접 URL 요청도 같은 기준으로 막히는지 확인했습니다.

  • 확인 조건 작성자가 아닌 사용자, 삭제된 게시글, 존재하지 않는 댓글
  • 검증 결과 실패 조건별 메시지와 응답 흐름을 문서에 기록

03 / Search & Input

검색, 페이징, 입력값 검증을 사용자 흐름으로 확인했습니다

제목, 내용, 작성자 검색 조건을 나누고 페이지 이동 후에도 조회 조건이 유지되는지 확인했습니다. 빈 제목, 긴 내용, 잘못된 검색 조건처럼 사용자가 만들 수 있는 입력 오류도 함께 점검했습니다.

  • 확인 조건 검색 조건 유지, 빈 값 입력, 길이 제한
  • 검증 결과 기능별 재현 방법과 수정 결과를 이슈 단위로 정리
