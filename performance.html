TFT-gogo에서 진행 중인 성능 개선 작업을 정리하는 페이지입니다. 개선 전후 측정 결과와 원인 분석, 적용한 방법을 함께 남길 예정입니다.

TFT-gogo

카탈로그 지연 로딩 전환으로 /api/guide 응답속도 87% 개선

avg latency -87% · TPS +639% (개선 전 대비)

개선 완료 · PR 병합 expand_more

Guide 페이지를 첫 번째 성능개선 대상으로 선정해 기준 성능을 측정하고 병목을 분석한 결과, 화면에는 쓰이지도 않는 카탈로그 전체 조회 API가 페이지 로드마다 불필요하게 호출되고 있다는 점이 근본 원인으로 확인됐습니다. 이를 없애는 방향으로 개선 방법을 확정해 코드로 옮겼고, 팀원이 작성한 코드가 PR로 병합되어 운영 코드 기준으로도 결과를 재검증했습니다.

GitHub Issue #647 보기 arrow_outward

문제 및 원인

부하 테스트(동시 사용자 50명, 10,100건 요청) 결과 GET /api/guide의 응답 크기(296.7 KB)가 다른 tab-items API 대비 8~30배 컸고, 스파이크 테스트에서는 이 구간의 응답 지연이 20ms → 570ms(28배)까지 치솟는 것을 확인했습니다. 프론트 소비 경로를 추적한 결과 이 API는 실제 화면 표시에는 쓰이지 않고 ① patchVersion 표시 ② 탭 조회 실패 시 fallback 원본 용도로만 매 페이지 로드마다 선호출되고 있었고, 실제 화면 데이터는 이미 페이지네이션이 적용된 GET /api/guide/{tab}에서 가져오는 중이었습니다.

✅ 채택한 개선 방법 — 카탈로그 지연 로딩 전환

  • · 백엔드: 전체 카탈로그 조회 없이 patchVersion만 반환하는 경량 엔드포인트 GET /api/guide/patch-version 신설
  • · 프론트: useGuideCatalog가 전체 카탈로그 대신 경량 엔드포인트만 호출하도록 전환
  • · fallback: 탭 조회 실패 시 라이브 카탈로그 대신 정적 번들(guideFallbackData)을 사용하도록 변경
  • · 기존 GET /api/guide 엔드포인트 자체는 다른 소비처를 위해 코드 그대로 유지

처음에는 원인을 DB 레이어로 의심해 팀원은 HikariCP 커넥션 풀 확장을, 본인은 Spring Cache(Caffeine) + 병렬 쿼리를 각각 조사했습니다. 둘 다 평균 응답은 개선됐지만 풀 확장은 p95/p99 꼬리 지연이 오히려 악화됐고, 캐싱은 애초에 필요 없는 호출을 빠르게 만드는 것일 뿐 근본 원인 해결은 아니라고 판단해 둘 다 미채택했습니다. 이후 프론트 소비 경로를 더 추적해 찾아낸, 호출 자체를 없애는 이 방법을 최종 채택했습니다.

배치 조회 + 캐싱 도입으로 /api/decks/meta N+1 해소, 응답속도 최대 99% 개선

avg latency 480ms → 5.8ms · TPS 103.3/s → 8,465.9/s (캐시 HIT 기준)

개선 완료 · PR 병합 expand_more

덱모음(메타 덱) 목록 조회에서 지연 로딩되는 4개 연관 컬렉션 때문에 덱 1개당 최대 3개씩 쿼리가 발생하는 N+1 구조를 확인했습니다. 배치 조회로 우선 해소했지만, 덱 수를 늘려가며 스케일 검증한 결과 배치 조회만으로는 데이터가 많아질수록 다시 느려지는 한계가 있어 캐싱까지 추가로 도입했습니다. 팀원이 작성한 코드가 PR로 병합되어 운영에 반영됐습니다.

GitHub Issue #709 보기 arrow_outward

문제 및 원인

GET /api/decks/meta는 MetaDeck의 4개 @OneToMany 연관관계(units, traits, heroAugments, artifactStats)를 지연 로딩하는데, 덱 1개당 최대 3개씩 개별 쿼리가 실행돼 덱 40건 조회 시 요청당 쿼리 수가 123개까지 쌓이는 N+1 구조였습니다. JMeter 부하 테스트(동시 사용자 50명 · 5분 · 3회 평균)에서 avg 480ms · TPS 103.3/s로 확인됐습니다.

✅ 채택한 개선 방법 — 배치 조회 + 응답 캐싱

처음에는 배치 조회만으로 해결을 시도했습니다. 4개 컬렉션에 @BatchSize(size = 100)을 적용해 덱 ID들을 IN절로 묶어 컬렉션 종류당 1쿼리로 배치 조회하도록 했습니다(join fetch는 MultipleBagFetchException·카테시안 곱 위험이 있어 배제). 팀원(gungang1212-tech)이 독립적으로 같은 방법을 제안해 함께 검증했고, 요청당 쿼리 수 123 → 6, avg 480ms → 112ms로 크게 개선됐습니다.

하지만 45 / 201 / 501덱으로 규모를 늘려 스케일 검증한 결과 배치 조회만으로는 덱 수가 늘어날수록 다시 느려지는 한계가 확인돼 근본 해결이 아니라고 판단, Caffeine 기반 캐싱(@Cacheable, TTL 1시간)을 추가로 도입했습니다. 메타 덱은 하루 1회 배치 집계 때만 바뀌는 준정적 데이터라 캐싱에 적합했고, 캐시 HIT 시에는 덱 수와 무관하게 거의 고정된 응답속도를 유지하며 MISS 시에도 배치 조회가 안전망 역할을 하도록 설계했습니다. 일일 집계 완료·관리자 큐레이션 변경 시에는 캐시를 전체 무효화해 stale 데이터 노출을 방지했습니다.

이후 코드 리뷰(PR #714)에서 캐시 무효화가 트랜잭션 커밋 이전에 실행돼 동시 조회 시 stale 데이터가 재캐싱될 수 있는 문제를 발견해 CacheManager를 TransactionAwareCacheManagerProxy로 감싸 보완했고, 집계 도중 예외가 발생해도 캐시가 무효화되도록 finally 블록에서 처리했습니다.

open-in-view 차단으로 /api/ai/recommend 커넥션 풀 경쟁 해소

avg latency -17% · TPS +20% (개선 전 대비) · 후속 조사는 비용 가드 문제로 보류

개선 완료 · PR 병합 expand_more

AI 덱추천(GET /api/ai/recommend)을 네 번째 성능개선 대상으로 선정해 기준 성능을 측정한 뒤, 단계별 타이밍 로그로 병목을 좁혀 DB 커넥션 풀 경쟁을 근본 원인으로 확정하고 개선했습니다. 이 개선을 적용하고 재측정하자 병목이 AI 서버(ai-server)의 단일 프로세스 CPU 포화로 이동한 것을 확인했는데, 이 후속 병목은 실측으로 효과를 검증하고도 비용 가드가 무력화되는 부작용 때문에 의도적으로 반영하지 않았습니다.

GitHub Issue #733 보기 arrow_outward

문제 및 원인

JMeter 부하 테스트(동시 사용자 50명 · 5분 · 3회 평균, OPENAI_API_KEY를 비운 fallback 경로 기준)에서 avg 429ms · p99 791ms · TPS 115.2/s로 확인됐습니다. 단계별 타이밍 로그를 추가해 원인을 좁힌 결과, spring.jpa.open-in-view가 기본값(true)으로 켜져 있어 요청 스레드가 AI 서버 동기 호출을 포함한 요청 처리 전체 동안 DB 커넥션을 쥐고 있는 것이 원인이었습니다. 50개 동시 요청이 HikariCP 기본 풀(10개)을 놓고 경쟁하면서, 요청당 첫 DB 접근인 getProfile() 단계가 쿼리 자체가 아니라 커넥션 획득 대기로 140~240ms를 잡아먹고 있었습니다.

✅ 채택한 개선 방법 — open-in-view 차단

  • · spring.jpa.open-in-view=false를 코드 레벨 기본값으로 설정 (application*.yml은 개인 설정이라 .gitignore 대상이라 코드 쪽에 둬야 모든 환경에 일관 적용됨)
  • · lazy-loading 회귀 위험 검토: 모든 컨트롤러가 엔티티가 아닌 DTO만 반환하고 엔티티→DTO 변환이 서비스 계층 트랜잭션 안에서 끝나는 패턴을 코드베이스 전수 확인 후 안전하다고 판단
  • · open-in-view를 끄고 재측정하니 AiServerClient가 JVM 기본 http.maxConnections=5에 걸려 다음 병목이 되는 것을 확인해 100으로 완화

🔍 후속 조사 — 효과는 확인했지만 반영은 보류

부하 중 docker stats로 확인한 결과, ai-server 컨테이너가 uvicorn worker 1개로만 떠 있어 CPU가 97.76%(코어 1개 포화)까지 치솟아 다음 병목으로 드러났습니다(Issue #736). worker 수를 2, 4로 늘려 탐색적으로 측정한 결과 성능이 거의 선형으로 개선됐습니다.

worker 수 avg p99 TPS ai-server CPU
1 (기존) 355ms 900ms 138.7/s 97.76%
2 278ms 614ms 177.1/s 196.99%
4 201ms 460ms 244.4/s 395.46%

worker=4에서 p99가 900ms → 460ms로 절반 이하로 줄어 가설(단일 worker CPU 포화가 꼬리 지연의 원인)은 확인됐지만, 그대로 merge하지 않았습니다. OpenAI 비용을 지키는 전역 rate limiter(30req/60s)와 서킷브레이커가 프로세스 메모리(모듈 전역변수)에 있어서, worker를 N개로 늘리면 이 상태가 워커마다 따로 생겨 "60초에 30번"이라는 비용 한도가 사실상 "60초에 30×N번"으로 느슨해집니다. 성능 수치만 보고 반영하면 의도치 않게 비용 가드를 무력화시키게 되어, 측정 결과만 이슈에 남기고 코드는 반영하지 않은 채 종료했습니다. rate limiter를 워커 간 공유하는 방법(Redis 이관 또는 워커당 예산 분할)이 정해지면 이 측정값을 근거로 재검토할 예정입니다.
