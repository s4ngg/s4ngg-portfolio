개선 방법 확정 · PR 병합 대기

Guide 페이지를 첫 번째 성능개선 대상으로 선정하고, 개선 전 기준 성능을 측정한 뒤 병목 지점을 좁혀 1개 단위로 개선하는 작업입니다. 팀원과 각자 다른 방식으로 원인을 조사하고 개선안을 테스트한 뒤, 근본 원인을 없애는 방식으로 개선 방법을 확정하고 개선 후 측정까지 마친 단계입니다. 실제 코드 변경 PR은 아직 병합 전이며, 병합되면 운영 코드 기준으로 다시 검증할 예정입니다.

✅ 채택한 개선 방법 — 카탈로그 지연 로딩 전환

프론트 소비 경로를 추적한 결과 GET /api/guide(4개 테이블 전체 조회 + JSON 직렬화)는 실제 화면 표시에는 쓰이지 않고, ① patchVersion 표시 ② 탭 조회 실패 시 fallback 원본 용도로만 매 페이지 로드마다 불필요하게 선호출되고 있었습니다. 실제 화면 데이터는 이미 페이지네이션이 적용된 GET /api/guide/{tab}에서 가져오고 있었습니다.

  • · 백엔드: 전체 카탈로그 조회 없이 patchVersion만 반환하는 경량 엔드포인트 GET /api/guide/patch-version 신설
  • · 프론트: useGuideCatalog가 전체 카탈로그 대신 경량 엔드포인트만 호출하도록 전환
  • · fallback: 탭 조회 실패 시 라이브 카탈로그 대신 정적 번들(guideFallbackData)을 사용하도록 변경
  • · 기존 GET /api/guide 엔드포인트 자체는 다른 소비처를 위해 코드 그대로 유지

함께 검토했던 다른 방법 (미채택)

1차 회의에서 "개선 방법 각자 1~2개 조사 후 취합" 하기로 한 대로, 팀원과 서로 다른 방향으로 개선안을 조사하고 실측했습니다.

팀원 조사 — HikariCP 커넥션 풀

maximum-pool-size가 기본값 10으로 방치되어 커넥션 획득 대기(hikari_pending 38~40)가 상시 발생 중임을 확인. 10 → 30 → 50으로 단계적으로 올리며 재검증.

pool=50에서 pending이 0으로 사라지고 avg 164ms → 134ms까지 개선됐지만, p95/p99(꼬리 지연)는 오히려 악화되고 CPU도 상승. 페이지네이션 없이 4개 테이블을 전부 조회하는 쿼리 자체가 남은 원인이라는 결론에 도달, 이 조사가 지연 로딩 전환 아이디어로 이어짐.

본인 조사 — Spring Cache(Caffeine) + 병렬 쿼리

Guide 데이터는 CDragon import 시에만 갱신되는 준정적 데이터라는 점에 착안해 @Cacheable/@CacheEvict 캐싱과 CompletableFuture 병렬 쿼리를 함께 적용.

avg 82ms → 12ms(-85%), TPS 330/s → 3,744/s(+1,035%)로 수치상 가장 좋았지만, 애초에 필요 없는 호출을 캐싱으로 빠르게 만드는 것일 뿐이라 근본 원인 해결이 아니라고 판단해 미채택.

두 방법 모두 유효한 개선이었지만, "더 이상의 기능 추가 없이 근본적인 문제를 해결하는 것이 맞다"는 판단 아래 불필요한 호출 자체를 없애는 카탈로그 지연 로딩 전환을 최종 채택했습니다.

회귀 테스트

카탈로그 지연 로딩 전환이 기존 5개 Guide 엔드포인트(catalog, traits, items, augments, champions)에 영향을 주는지 baseline과 동일 조건(50명 × 20루프)으로 2회 재검증했습니다. 1차 실행에서 p99만 균일하게 급등했으나 서버 재시작 직후 JVM 워밍업 아티팩트로 확인됐고, 2차 실행에서는 baseline과 동등하거나 더 나은 수치가 나와 회귀 없음을 확정했습니다.

다음 단계
