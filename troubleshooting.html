트러블슛팅은 코드를 고친 결과뿐 아니라 문제가 발생한 프로젝트, 재현 조건, 원인을 좁힌 과정, 수정 후 다시 확인한 기준까지 함께 남기는 기록이라고 생각합니다. 아래 사례는 단순 해결 요약이 아니라 로그, 요청 흐름, 트랜잭션 경계, 쿼리 실행 방식까지 확인하며 원인을 좁혀간 예시입니다.

미니프로젝트 · AllPick

역할 기반 JWT 인증 흐름에서 Role 정보 불일치가 발생한 문제

JWT · Redis · Spring Security
Problem
AllPick은 구매자, 판매자, 관리자가 함께 사용하는 쇼핑몰 플랫폼입니다. QA 과정에서 로그인은 성공했지만 이후 일부 요청에서 사용자 역할이 정상적으로 인식되지 않아, 역할별로 접근해야 하는 API 흐름이 흔들리는 문제가 발생했습니다. 판매자 승인, 상품 승인, 주문/환불 처리처럼 권한 기준이 명확해야 하는 기능에서 토큰과 역할 정보가 맞지 않으면 잘못된 인증 상태로 요청이 처리될 수 있었습니다.
Cause
처음에는 로그인/회원가입 API 응답 문제이거나 JWT 보안 설정 문제라고 판단해 인증 API와 토큰 검증 로직을 먼저 확인했습니다. 토큰 저장 방식, 요청 헤더, Spring Security 설정을 수정해보았지만 문제가 해결되지 않았고, 정상 로그인 이후에도 특정 역할이 계속 인식되지 않았습니다. 이후 로그인 응답과 회원 도메인 변경 이력을 함께 추적하면서, 초기 설계에서는 member에 role을 부여했다가 이후 제거했는데 기존 로그인 코드에는 role을 기준으로 처리하는 로직이 남아 있다는 점을 발견했습니다. 결과적으로 현재 회원 데이터 구조와 로그인 코드의 역할 처리 방식이 서로 맞지 않아 JWT 인증 이후 권한 판단이 어긋난 것이 원인이었습니다.
Solution
먼저 로그인 코드에서 더 이상 사용하지 않는 member role 의존 로직을 제거하고, 실제 서비스 구조에 맞게 일반 회원, 판매자, 관리자 인증 흐름을 분리했습니다. 프론트엔드에서는 userToken, sellerToken, adminToken을 구분해 저장하고 API 경로에 따라 올바른 토큰을 Authorization 헤더에 실도록 정리했습니다. 서버에서는 JWT의 권한 정보와 요청 대상 API의 역할 기준을 다시 맞췄고, Refresh Token은 Redis에 저장하며 로그아웃 또는 토큰 폐기 시 블랙리스트 처리로 기존 토큰이 계속 사용되지 않도록 보완했습니다.
Result
구매자, 판매자, 관리자 계정으로 각각 로그인한 뒤 상품 등록, 판매자 승인, 상품 승인, 주문/환불 처리 API를 다시 호출하며 접근 가능 범위를 검증했습니다. 잘못된 역할의 토큰으로 요청했을 때는 권한 오류가 반환되고, 올바른 역할의 토큰으로 요청했을 때만 기능이 동작하는 것을 확인했습니다. 이 과정을 통해 화면 상태가 아니라 서버 권한 검증을 기준으로 역할별 흐름을 안정화했습니다.
