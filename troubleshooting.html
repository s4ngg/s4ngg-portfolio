문제를 재현하고 원인을 좁힌 기록

트러블슛팅은 코드를 고친 결과뿐 아니라 문제가 발생한 프로젝트, 재현 조건, 원인을 좁힌 과정, 수정 후 다시 확인한 기준까지 함께 남기는 기록이라고 생각합니다. 아래 사례는 단순 해결 요약이 아니라 로그, 요청 흐름, 트랜잭션 경계, 쿼리 실행 방식까지 확인하며 원인을 좁혀간 예시입니다.

미니프로젝트 · AllPick

역할별 요청이 섞이며 권한 검증이 흔들리는 문제

JWT · Redis · Spring Security
Problem
AllPick은 구매자, 판매자, 관리자가 함께 사용하는 쇼핑몰 플랫폼입니다. 기능 구현 이후 QA 과정에서 관리자 API, 판매자 API, 일반 회원 API 요청이 같은 인증 흐름으로 처리되며 역할별 접근 오류가 발생할 수 있다는 점을 확인했습니다. 예를 들어 판매자 승인, 상품 승인, 주문/환불 처리처럼 역할이 명확히 나뉘어야 하는 기능에서 토큰 저장 위치와 요청 헤더 기준이 섞이면 잘못된 권한으로 요청이 전달될 위험이 있었습니다.
Cause
처음에는 단순히 화면 버튼 노출이나 라우팅 문제로 볼 수 있었지만, 실제 원인은 클라이언트와 서버 사이의 인증 계약이 역할별로 충분히 분리되지 않은 데 있었습니다. Spring Security는 토큰의 권한을 기준으로 API 접근을 판단하지만, 프론트엔드에서 userToken, sellerToken, adminToken의 사용 기준이 모호하면 요청 단계에서 잘못된 토큰이 포함될 수 있었습니다. 또한 로그아웃, 토큰 만료, 재로그인 상황에서 기존 토큰이 남아 있으면 서버 권한 검증과 화면 상태가 어긋날 수 있었습니다.
Solution
역할별 토큰을 userToken, sellerToken, adminToken으로 분리하고, API 경로에 따라 어떤 토큰을 Authorization 헤더에 실을지 명확히 나눴습니다. 서버에서는 JWT 권한 정보를 기준으로 관리자, 판매자, 일반 회원 접근 범위를 다시 검증하도록 정리했습니다. Refresh Token은 Redis에 저장하고, 로그아웃 또는 토큰 폐기 시 블랙리스트 처리 흐름을 두어 사용자가 로그아웃한 뒤 기존 Access Token이 계속 사용되지 않도록 했습니다.
Result
구매자, 판매자, 관리자 계정으로 각각 로그인한 뒤 상품 등록, 판매자 승인, 상품 승인, 주문/환불 처리 API를 다시 호출하며 접근 가능 범위를 검증했습니다. 잘못된 역할의 토큰으로 요청했을 때는 권한 오류가 반환되고, 올바른 역할의 토큰으로 요청했을 때만 기능이 동작하는 것을 확인했습니다. 이 과정을 통해 화면 상태가 아니라 서버 권한 검증을 기준으로 역할별 흐름을 안정화했습니다.
